S – Single Responsibility Principle

1. Single Responsibility Prinzip

Aufgabe: Was ist das Problem an dieser Klasse?
class Report {
    void generate() {
        // Daten für den Bericht sammeln und formatieren
    }
    void print() {
        // Bericht auf dem Drucker ausgeben
    }
    void exportToPdf() {
        // Bericht als PDF-Datei exportieren
    }
}

Lösung

Diese Klasse verletzt das Single Responsibility Principle, weil sie mehrere Verantwortlichkeiten besitzt:

  • Bericht erzeugen
  • Bericht drucken
  • Bericht als PDF exportieren

Eine Klasse sollte nur einen Grund zur Änderung haben. 

class Report {
    void generate() {
        // Bericht erzeugen
    }
}

class ReportPrinter {
    void print(Report report) {
        // Bericht drucken
    }
}

class ReportPdfExporter {
    void exportToPdf(Report report) {
        // Bericht als PDF exportieren
    }
}
O – Open/Closed Principle

2. Open/Closed Prinzip

Aufgabe: Warum verletzt diese Klasse das OCP?
class ShippingCalculator {
    public double calculate(String country) {
        if (country.equals("Austria")) {
            return 4.90;
        } else if (country.equals("Germany")) {
            return 6.50;
        } else {
            return 10.00;
        }
    }
}

Lösung

Diese Klasse verletzt das Open/Closed Principle, weil sie bei jeder Erweiterung geändert werden muss. Wenn ein neues Land hinzukommt, muss die Methode angepasst werden.

Klassen sollten für Erweiterung offen, aber für Änderung geschlossen sein. 

interface ShippingStrategy {
    double calculate();
}

class AustriaShipping implements ShippingStrategy {
    public double calculate() {
        return 4.90;
    }
}

class GermanyShipping implements ShippingStrategy {
    public double calculate() {
        return 6.50;
    }
}

class DefaultShipping implements ShippingStrategy {
    public double calculate() {
        return 10.00;
    }
}

class ShippingCalculator {
    public double calculate(ShippingStrategy strategy) {
        return strategy.calculate();
    }
}
L – Liskov Substitution Principle

3. Liskovsches Substitutionsprinzip

Aufgabe: Warum ist diese Vererbung problematisch?
class Account {
    void withdraw(double amount) { ... }
    void deposit(double amount) { ... }
    double getBalance() { ... }
}

class ReadOnlyAccount extends Account {
    void withdraw(double amount) {
        throw new UnsupportedOperationException();
    }
    void deposit(double amount) {
        throw new UnsupportedOperationException();
    }
    double getBalance() {
        return ...;
    }
}

Lösung

Das verletzt das Liskov Substitution Principle, weil ein ReadOnlyAccount kein echter Ersatz für ein normales Account ist.

Wer ein Account erwartet, rechnet damit, dass withdraw() und deposit() funktionieren. Beim ReadOnlyAccount kommt aber eine Exception. 

interface ReadableAccount {
    double getBalance();
}

interface WritableAccount extends ReadableAccount {
    void withdraw(double amount);
    void deposit(double amount);
}

class Account implements WritableAccount {
    public void withdraw(double amount) { }
    public void deposit(double amount) { }
    public double getBalance() { return 0; }
}

class ReadOnlyAccount implements ReadableAccount {
    public double getBalance() { return 0; }
}
I – Interface Segregation Principle

4. Interface Segregation Prinzip

Aufgabe: Was passt an diesem Interface nicht?
interface Vehicle {
    void drive();
    void fly();
    void sail();
}

class Car implements Vehicle {
    public void drive() {
        System.out.println("Fährt auf der Straße.");
    }
    public void fly() {
        throw new UnsupportedOperationException("Ein Auto kann nicht fliegen!");
    }
    public void sail() {
        throw new UnsupportedOperationException("Ein Auto kann nicht segeln!");
    }
}

Lösung

Das verletzt das Interface Segregation Principle, weil das Interface zu groß ist. Eine Klasse wird gezwungen, Methoden zu implementieren, die sie gar nicht braucht.

Ein Auto kann fahren, aber nicht fliegen oder segeln. 

interface Drivable {
    void drive();
}

interface Flyable {
    void fly();
}

interface Sailable {
    void sail();
}

class Car implements Drivable {
    public void drive() {
        System.out.println("Fährt auf der Straße.");
    }
}
D – Dependency Inversion Principle

5. Dependency Inversion Prinzip

Aufgabe: Warum verletzt diese Klasse das DIP?
class EmailSender {
    public void send(String message) {
        System.out.println("E-Mail gesendet: " + message);
    }
}

class NotificationService {
    private final EmailSender sender = new EmailSender();

    public void notify(String message) {
        sender.send(message);
    }
}

Lösung

Das verletzt das Dependency Inversion Principle, weil NotificationService direkt von der konkreten Klasse EmailSender abhängt.

Besser ist die Abhängigkeit von einer Abstraktion, etwa einem Interface. 

interface MessageSender {
    void send(String message);
}

class EmailSender implements MessageSender {
    public void send(String message) {
        System.out.println("E-Mail gesendet: " + message);
    }
}

class NotificationService {
    private final MessageSender sender;

    public NotificationService(MessageSender sender) {
        this.sender = sender;
    }

    public void notify(String message) {
        sender.send(message);
    }
}
